## Codex 行为约束

- **禁止重复打开浏览器**：`open` 命令只需执行一次。如果 `open` 返回 exit code 0 或已执行过，不应再次调用。如果第一次没有弹出窗口，改用 `echo` 提示用户手动打开 URL。
- **禁止死循环修复**：同一个 bug 修复后验证不超过 3 次。如果修复 3 次仍未解决，应停下来分析根因而非继续尝试。
- **工具调用上限**：同一工具（如 `open`）在同一轮对话中连续调用不超过 **2 次**。

## 一、产品概述

### 1.1 产品名称
**测试用例记录表**（TestCase Viewer & Recorder）

### 1.2 一句话描述
将 Excel 中的测试用例逐条展示为清晰易读的 HTML 页面，支持测试人员逐条执行、记录结果、回写 Excel。

### 1.3 解决的问题
测试用例通常包含大量字段（前置条件、步骤、预期结果、优先级等），在笔记本屏幕上一行显示全部信息会导致：
- 文字被压缩、换行混乱，阅读困难
- 测试人员难以快速定位「我要测什么、怎么判断通过」
- 执行过程中容易遗漏关键验收点

本工具将**一条用例 = 一个独立页面**，只展示当前这条用例的完整信息，让测试人员逐条专注执行。

## 三、核心功能

### 3.1 三个视图（顶部标签页切换）

**① 执行测试** — 逐条展示用例卡片
- 每条用例独立卡片：编号、标题、模块、优先级、前置条件、测试步骤（自动编号）、预期结果（绿色高亮）、测试目的（黄色高亮）
- 测试结果选择：通过 / 失败 / 阻塞 / 跳过（颜色区分）
- 测试现象备注文本框
- 「确认并保存」按钮 → 结果写入 Excel
- 进度条 + 图例（通过/失败/阻塞/跳过/未执行）
- 键盘快捷键：← → 切换用例，Ctrl+S 保存

**② 搜索筛选** — 快速定位目标用例
- 全文关键词搜索（搜索所有字段内容）
- 结果筛选：通过/失败/阻塞/跳过/未执行
- 优先级筛选
- 模块筛选
- 搜索结果关键词高亮，点击直接跳转到执行视图
- 分页支持（大量用例时）

**③ 汇总统计** — 全局执行状态一览
- 统计卡片：总数、通过、失败、阻塞、跳过
- 环形进度图：执行进度百分比、通过率百分比
- 按优先级分布表（含进度条）
- 按模块分布表（含进度条）

### 3.2 读取测试用例 Excel
- 用户将测试用例 Excel 放在 `testcases/` 目录下
- 程序中自动读取第一个 Excel 文件的第一个工作表
- 支持的 Excel 格式：`.xlsx`、`.xls`

### 3.3 结果回写 Excel
- 点击「确认」后，将测试结果和备注写入 Excel 对应行
- 写入列：`测试结果`、`测试现象备注`（如 Excel 中无此列则自动新增）
- 同时记录执行时间
- 内存状态实时更新，无需重新读取 Excel 即可在搜索和汇总页看到最新结果

> 以下三个方案都能满足核心需求（读取 Excel → HTML 展示 → 记录结果 → 回写 Excel → 跨平台）。区别在于**架构复杂度、界面美观度、启动方式**。

### 方案一 🌐：Python Flask + 浏览器前端 【推荐】

**怎么工作的：**
```
你双击 start → Python 启动一个微型网站（只在你自己电脑上）→ 自动打开浏览器 → 页面展示用例
```

**优点：**
- 界面最美观，可以做出专业级 UI（卡片阴影、渐变色、动画过渡）
- Flask 是 Python 最成熟的 Web 框架，稳定可靠
- 前后端分离，以后想加功能（比如搜索、统计页、导出报告）很方便
- CSS 可以充分利用浏览器能力（打印、缩放、响应式）

**缺点：**
- 需要安装 2 个 pip 包（Flask、openpyxl），启动脚本会自动装，但首次要多等几秒
- 需要同时开着浏览器和终端窗口

**适合你吗？**
→ 如果你对界面有要求、希望看起来专业、以后可能会加功能，选这个。

---

### 方案二 🪟：Python tkinter 桌面应用

**怎么工作的：**
```
你双击 start → 弹出一个原生桌面窗口 → 窗口里展示用例 → 在窗口内操作
```

**优点：**
- **零额外依赖**，tkinter 是 Python 自带的，不需要 pip 装任何东西
- 不依赖浏览器，一个独立窗口完成所有操作
- 最像"传统软件"的体验，对不熟悉浏览器的用户友好
- 可以通过系统文件对话框直接选择 Excel 文件

**缺点：**
- 界面较"土"，像 Windows 2000 年代的程序风格（虽然可以用 ttk 主题稍微美化）
- 布局灵活性不如网页，字段多了会挤
- 以后加功能（搜索、统计图表）比较费劲

**适合你吗？**
→ 如果你追求"最简单、最少依赖、点开就用"，选这个。

---

### 方案三 📄：纯静态 HTML + 轻量 Python 后端

**怎么工作的：**
```
你双击 start → Python 读取 Excel 转为 JSON → 启动一个极简 HTTP 服务 → 打开一个自包含的 HTML 页面 → 页面用 JS 展示用例
```

**优点：**
- 架构最清晰：Python 只管"读 Excel、写 Excel"，HTML 只管"展示界面"
- HTML 文件是纯静态的，你可以用浏览器直接打开预览（只是数据需要 Python 喂）
- 代码量最少，出问题最好排查
- HTML 文件可以单独发给同事看效果

**缺点：**
- 每个操作（保存结果、切换用例）都需要 HTTP 请求，比方案一略慢（几乎感觉不到）
- 仍然需要 Python 做后端（不能纯双击 HTML 运行，因为浏览器无法直接写 Excel）

**适合你吗？**
→ 如果你追求"代码最简单、最透明、最好维护"，选这个。

---

### 我的推荐：方案一 🌐

理由很简单：**你的核心痛点是"屏幕上信息太多看不清"，浏览器能给你最好的阅读体验**——大字体、卡片布局、颜色标记这些都是浏览器的强项。Flask 够稳定，两个依赖包启动脚本自动搞定，你不需要操心。

## 六、Excel 格式约定

### 6.1 自动识别规则
- 第一行为表头（列名）
- 从第二行开始为用例数据
- 脚本自动识别常见列名并映射到展示区域：
  - 用例编号：`编号`、`ID`、`用例ID`、`序号`
  - 用例标题：`标题`、`用例名称`、`测试点`
  - 前置条件：`前置条件`、`预置条件`
  - 测试步骤：`步骤`、`测试步骤`、`操作步骤`
  - 预期结果：`预期结果`、`期望结果`
  - 测试目的：`测试目的`、`目的`、`测试目标`
  - 优先级：`优先级`、`等级`
- 未识别的列名也会展示在「其他信息」区域

### 6.2 回写列
- 如果 Excel 中不存在 `测试结果` 和 `测试现象备注` 列，会自动在表头新增
- 执行时间写入 `执行时间` 列

## 七、界面设计要点

1. **大字体、高对比度**：适合长时间阅读
2. **卡片式布局**：信息分组清晰，每条用例一个视觉单元
3. **响应式宽度**：适配 13-16 寸笔记本屏幕
4. **进度条**：直观显示当前执行进度
5. **颜色标记**：
   - 通过 → 绿色
   - 失败 → 红色
   - 阻塞 → 橙色
   - 跳过 → 灰色
   - 未执行 → 默认色待定

## 九、非功能性需求

1. **零配置启动**：用户只需双击启动脚本，无需手动安装任何东西
2. **中文支持**：Excel 内容、界面文案均为中文
3. **错误处理**：Excel 格式异常时给出友好的中文提示
4. **性能**：支持 1000+ 条用例流畅切换

## 十、约束与假设

1. 假设用户电脑已安装 Python 3（Mac 默认已安装；Windows 需先安装，启动脚本会检测并提示）
2. 假设测试用例 Excel 格式规整（第一行表头，下方数据行）
3. 假设浏览器为 Chrome / Edge / Safari 等现代浏览器

## 十一、开发流程规则（vibe-coding 规范化方案落地）

> 以下规则确保开发过程遵循规范化方案，**AI 必须严格遵守**。

### 1. 新功能需求 → 必须先走 PRD 流程

用户提出新功能需求时（如"我要加一个 XXX 功能"），**必须**：
1. 先加载 `requirement-to-prd` skill（路径：`~/.workbuddy/skills/requirement-to-prd/SKILL.md`）
2. 按其流程：初步解析 → 逐步提问（一次 1-2 个）→ 写 PRD → 用户确认
3. **PRD 确认后才能开始写代码**，不确认不动手

### 2. 提交前检查 → pre-commit hook 自动拦截

每次 `git commit` 前自动跑：JS 语法 + Python 语法 + `make test`（52 passed）。不通过阻止提交。

### 3. 分支策略 → 不在 main 上直接改

新功能建 `feat/` 分支，重构建 `refactor/` 分支，修 bug 建 `fix/` 分支。main 只接收合并。

### 4. 测试先行 → 改代码后必跑测试

任何代码改动后必须跑 `make test` 确认不回归。测试红了就是改坏了，测试绿了才安全。

### 5. 编码规范 → Conventional Commits + 代码检查

- 提交信息用规范格式：`feat:` / `fix:` / `style:` / `refactor:` / `docs:` / `chore:`
- `make lint` 检查 Python 语法 + JS 语法（工具未装时降级为 py_compile + node --check）
