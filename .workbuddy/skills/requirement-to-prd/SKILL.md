# 需求转 PRD 工作流

> 当用户提出新功能需求时，进入此工作流。目标：把模糊的想法变成清晰的 PRD 文档。

## 工作流程

### 第一步：初步解析（用户说完需求后立即做）

用户提出需求后，AI 先解析出三件事，用大白话告诉用户：

1. **功能一句话描述**：我理解你要做的是 XXX，对吗？
2. **初步实现方案**：技术栈、后端 API、数据模型（简要）
3. **初步 UI 设计方案**：页面布局、组件、交互方式（简要）

然后说："我还有几个问题想确认一下，帮你把需求理清楚。"

### 第二步：逐步提问（关键：一次只问 1-2 个问题）

按以下顺序逐步提问。**每轮只问 1-2 个问题**，等用户回答后再问下一轮。不要一次抛 5 个问题。

#### 第 1 轮：功能价值
- 这个功能主要解决什么问题？
- 谁会用这个功能？什么场景下会用？

#### 第 2 轮：输入输出
- 用户需要输入什么？或者数据从哪来？
- 功能输出什么？（展示在页面上 / 导出文件 / 存到数据库）

#### 第 3 轮：交互细节
- 怎么触发这个功能？（点按钮 / 自动执行 / 定时执行）
- 执行完有没有状态变化或通知？

#### 第 4 轮：边界条件
- 数据为空或出错时怎么处理？
- 有没有权限限制？（谁能看谁能改）

#### 第 5 轮：优先级
- 这个功能的优先级？（P0 必须 / P1 重要 / P2 可选）
- 有没有依赖的其他功能？

> **注意**：不是每个问题都要问。如果用户在描述需求时已经说了，就跳过。只问信息缺失的部分。

### 第三步：写 PRD

信息补充完整后，写进 PRD 文档：

- **文件路径**：项目 `docs/` 目录下，命名 `PRD-{功能名}.md`
- **写完后**：展示给用户确认，问"这个 PRD 有要补充或修改的吗？"
- **确认后**：问"要开始开发吗？"

## PRD 模板

```markdown
# PRD: {功能名}

> 版本: 1.0 | 日期: {YYYY-MM-DD} | 状态: 待开发

## 一、需求背景

- **问题**：{解决什么问题}
- **价值**：{功能的作用，为什么要做}
- **目标用户**：{谁会用}
- **使用场景**：{什么场景下用}

## 二、功能描述

- **一句话描述**：{功能做什么}
- **输入**：{用户输入什么 / 数据从哪来}
- **输出**：{功能输出什么}
- **触发方式**：{按钮 / 自动 / 定时}

## 三、实现方案

- **技术栈**：{后端语言 / 框架 / 数据库}
- **后端 API**：
  | 接口 | 方法 | 路径 | 作用 |
  |------|------|------|------|
  | {API名} | {GET/POST} | {/api/xxx} | {做什么} |
- **数据模型**：
  | 字段名 | 类型 | 说明 |
  |--------|------|------|
  | {字段} | {类型} | {说明} |
- **依赖**：{依赖的其他功能 / 模块}

## 四、前端 UI 设计

- **页面布局**：{描述布局结构}
- **组件**：{用哪些组件，如卡片/表格/表单/弹窗}
- **交互方式**：{用户怎么操作，步骤}
- **状态变化**：{有无动画 / 状态切换 / 通知}

## 五、边界条件

- **正常流程**：
  1. {步骤 1}
  2. {步骤 2}
  3. {步骤 3}
- **异常处理**：
  | 异常情况 | 处理方式 |
  |----------|----------|
  | {异常 1} | {怎么处理} |
- **权限**：{有无限制}

## 六、验收标准

- [ ] {标准 1：可验证的具体条件}
- [ ] {标准 2}
- [ ] {标准 3}

## 七、优先级

- **等级**：{P0 必须 / P1 重要 / P2 可选}
- **预期上线**：{时间或版本}

## 八、变更记录

| 日期 | 版本 | 变更内容 |
|------|------|----------|
| {日期} | v1.0 | 初版 |
```

## 关键原则

1. **一次问 1-2 个问题** — 不要一次问 5 个，用户会累
2. **用大白话问** — 不要用技术术语（用户可能是小白）
3. **每轮等用户回答** — 不要假设答案，不要替用户做决定
4. **信息不足时主动追问** — 宁可多问一轮，不要漏掉关键信息
5. **PRD 写完让用户确认** — 不要直接开始开发，先确认需求
6. **实现方案和 UI 设计要具体** — 不要写"待定"，给出具体建议让用户确认或调整

## 示例对话

```
用户：我想加一个批量导入测试用例的功能。

AI：好的，我先理解一下——你想做一个功能，让用户能一次性导入多条测试用例（比如从 Excel 文件批量上传），而不是一条一条手动输入。对吗？

初步方案：
- 实现：后端加 /api/import 接口，接收 Excel 文件，解析后批量写入
- UI：上传按钮 + 文件选择 + 导入结果反馈

我还有几个问题想确认：

1. 这个功能主要解决什么问题？是现在手动一条条输入太慢了吗？
2. 导入的数据来源是什么？用户自己整理的 Excel？还是从其他系统导出的？

用户：对，现在手动输入太慢了。数据来源是我们自己整理的 Excel。

AI：明白了。再确认两个问题：

1. 导入后如果遇到重复的用例（编号已存在），怎么处理？覆盖？跳过？还是报错让用户选？
2. 导入成功后需要反馈什么？导入了几条、成功几条、失败几条？

用户：重复的跳过就行。导入后告诉我成功几条失败几条。

AI：清楚了。最后确认一下优先级——这个功能是 P0 必须尽快做，还是 P1 重要但不急？

用户：P1 吧，下周做也行。

AI：好的，需求都清楚了，我写进 PRD。

（写 PRD 文件，展示给用户确认）
```
